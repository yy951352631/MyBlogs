NoSql

背景

在海量用户高并发的情况下，传统的关系型数据库性能达到瓶颈，尽管当下可以使用ShardingJDBC将数据分布在不同的服务器，在一定程度上缓解磁盘IO的压力，但当随着据量继续增加，这种方式对性能的提升也逐渐陷入瓶颈。

关系型数据库：数据关系复杂，拓展性差，不便于大规模集群

解决思路

1.降低磁盘IO次数，越低越好 —–外存存储–>内存存储

2.去除数据间关系，越简单越好 —–不存储关系，仅存储数据

NoSql（Not-Only SQL）作为非关系型数据库的补充（数据本质上还是存储在磁盘中）

NoSQL的特征

1.可扩容，可伸缩

2.大数据量下的高性能

3.灵活的数据模型

4.高可用

常见的NoSql数据库

1.Redis

2.memcache

3.HBash

4.MongoDB

Redis

简介

redis（REmote DIctionary Server（远程字典服务））是C语言开发的开源的高性能的键值对数据库。

特征

1.数据间没有必然的关联关系

2.内部采用单线程机制进行工作

3.高性能

4.多数据类型支持

5.持久化支持，可以进行数据灾难恢复。

应用

1.为热点数据加速查询

2.任务队列，如秒杀，抢购，购票排队等

3.即使信息查询。如各种排行榜，统计网站，公家车到站信息，在线人数（聊天室，网站）设备信号等。

4.时效性信息控制，如验证码控制，投票控制。

5.分布式数据共享，如分布式集群架构中的session分离

6.消息队列

7.分布式锁

单数据与多数据操作的区别

多数据操作可以节约redis与服务器之间的往返时间，提高缓存的效率，但是由于redis单线程的运作方式，如果一次性发送的数据量过大，会造成阻塞，因而我们需要平衡好发送的数据量与发送的次数之间的关系，找到效率较高的那个点。

Hash类型操作的注意事项

1.hash类型下的value只能存储字符串，不允许存储其他的数据类型，不存在嵌套现象，如果数据未获取到，对应值为nil

2.hash的存储是有上限的，每个hash可以存储2^32-1个键值对

3.hash类型十分贴近对象的数据存储形式，并且可以灵活的添加删除对象属性，但hash设计的初衷不是为了存储大量对象而设计的，切记不可滥用，更不可以将hash作为对象列表使用

4.hgetall操作可以获取全部属性，但是如果内部的field过多，遍历整体数据笑了就会很低，有可能成为数据访问的瓶颈。（需要慎用，尽量用到什么拿什么）

电商购物车设计与实现

解决方案

1.以顾客的id作为key，每位客户创建一个hash存储结构对应购物车信息

2.将商品编号作为field，购买数量作为value进程存储。

问题

（购物车：hmset user001 g001 2 g002 1 g003 1）

采用上述方案仅仅是将数据存储到了redis中，并没有起到加速的作用，商品信息实际中还需要到数据库中进行二次查询。

解决方案

1.每条购物车中的商品记录保存为两条field

2.field1专用于保存购买数量，命名格式：商品id：nums

3.field2专门用于保存购物车中显示的信息，包含文字描述，图片地址，所属商家信息等，命名格式：商品id：info，保存格式：json

一个问题

一个商品信息可能多次出现在多个用户的购物车中，有大量的重复数据

解决方案

将商品信息独立于当前的hash，重新建立一个hash进行存储（商品信息仍存储在redis中，且避免了重复存储）

tips

使用hsetnx key field value指令，如果当前哈希中有输入的field，则不会修改value值，否则会新增一个field并存入value。（如张三购买的商品1，商品1入到redis中，李四也购买了商品1，这时会涉及到重复添加商品的操作，可以使用以上指令，相当于在插入数据前先进行一个判断，没有才进行插入）

业务场景2

基础服务+增值服务类型网站会设定会员的使用，让用户充分体验会员的优势，那么我们该如何管理这些有时效性的信息呢？

解决方案

1.对于基于时间线限定的任务处理，将处理时间记录为score值，利用排序功能区分处理的先后顺序

2.记录下一个要处理的时间，到期后处理对应任务，移除redis中的记录并记录下一个要处理的时间

3.当新任务加入时，判定并更新当前下一个要处理的任务时间

4.为提升sorted_set的性能，通常将任务根据特征存储为若干个sorted_set。例如1小时内，1天内，1周内，1月内，季内，年度内等，操作时逐级提升，将即将操作的若干个任务纳入到处理的队列中。

redis - sorted_set可以应用于定时任务执行顺序管理或任务过期管理。

业务场景1

人工智能领域的语义识别自动对话将会是未来服务业机器人应答呼叫体系中的重要技术，百度自研用户评价语义识别服务，免费开放给企业使用，同时训练百度自己的模型，现对试用用户的使用行为进行限速，限制每个用户每分钟最多发起10次调用

解决方案

1.设计计数器，记录调用次数，用于控制业务执行次数，以用户id作为key，使用次数作为value

2.在调用前获取次数，判断是否超过限定次数，不超过次数的情况下每次调用次数+1，业务调用失败（回滚）计数-1

3.为计数器设置生命周期为指定周期（这里是一分钟），自动清空周期内使用次数

解决方案改良

1.取消最大值判定，利用incr操作超过最大值抛出异常的形式替代每次判断是否大于最大值

2.判断是否为nil，如果是，设置次数为Max-次数，如果不是，计数器+1，业务调用失败，计数器-1

3.遇到异常+操作超过上限，视为达到次数上限

（最大值：9223372036854775807）

业务场景2

接收微信消息时，当有多个好友同时发送消息时，该排序会不停的进行交替，同时还会将重要的会话设置为置顶，一旦用户离线后再次打开微信，消息该按照什么样的顺序显示？

解决方案

1.依赖list的数据具有顺序的特征，对消息进行管理，将list结构作为栈使用

2.对置顶与普通会话分别建立独立的list分别进行管理

3.当某个list接收到用户消息后，将消息发送方的id从list一侧加入到list

4.多个相同id发出消息反复入栈会出现问题，因此在入栈之前无论是否具有当前会话id对应的消息，都先删除对应id

5.推送消息时先推送置顶会话list，再推送普通会话list，推送完成的list清除所有的数据

6.消息的数量，也就是微信用户对话的数量采用计数器的思想另行记录，伴随list操作同步更新
